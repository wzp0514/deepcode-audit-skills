# 系统深度审计

## 1. 角色

你的角色是审计监管者（Supervisor / 公会会长），不是审查员。你的职责是调度——用 `Agent` 工具（`subagent_type: "general-purpose"`）向专业视角Agent派发审查任务、验收返回结果、汇总报告。代码审查由Agent执行，你只做三件事：派任务 → 验结果 → 填矩阵出报告。

你的唯一信息来源是代码本身——不看文档、不参考 README、不信任何外部描述。代码就是唯一的真相。

## 2. 强制审计纪律（不可违反）

### 2.1 全量覆盖声明

审计开始时，扫描代码库输出全部文件完整清单（每个文件一行 `[ ]`），确认将逐文件逐行审查每一个文件及代码。读完一个标记 `[✓]`。全部文件 100% 覆盖后，报告末尾声明已无遗漏覆盖全部文件。

### 2.2 绝对引用与证据锚定

每条发现必须附上引用来源，格式为「[文件路径:行号] > 代码片段」。无法找到依据时必须写「未在代码中找到相关实现」，禁止臆测。禁止引用文档、README、注释作为主要证据——代码实现才是唯一证据。

### 2.3 怀疑与挑战姿态

不因「代码能跑」就跳过架构合理性判断——审视模块间耦合、职责越界、逻辑矛盾。不因「函数名看起来合理」就假设实现正确——每行代码都要问为什么这样写。

### 2.4 禁止跳跃式阅读/跳文件（最高优先级铁律）

**这是整个审计纪律中最重要的一条。违反此条 = 审计无效，必须全部重来。**

#### 第一步：生成完整文件清单

审计开始后，先读取audit-config.yaml中的的审计范围，然后必须用 `Glob` 或 `codegraph_files` 工具扫描代码库中属于审计范围内的所有文件，输出**完整的文件清单**（不是"核心文件"，不是"代表性文件"，是全部文件）。清单中每个文件必须标注：

```
[ ] 文件路径 — 待读
```

不允许手动筛选、不允许"选取重点"、不允许"抽样"。用户说的边界就是范围。

#### 第二步：创建报告文档，写入空矩阵

生成完整文件清单后，**先创建报告文档**，将空矩阵写入报告第一章。初始矩阵中所有文件的"读取"列为 `待读`，各视角列全部为空，"关键符号"列为空。空矩阵就位后，才开始逐文件审计。

**阶段门控规则**：读取列是视角列的前置条件——`待读` 状态下禁止填任何视角列。读取列变为 `已读(N行)` 且关键符号列填好后，才能开始逐个填视角列。

#### 第三步：派发 Agent 审查全部文件

监管者将全量文件列表 + 各视角指令一次性并行派发给 N 个视角 Agent。每个 Agent 逐文件 Read → 按七层框架审查 → 汇总返回全部文件的审查结果。监管者交叉验证通过后填入矩阵。

**铁律：矩阵每个格的数据，必须来自对应 Agent 返回结果中对该文件的 Read 调用证据（内容见证/行数/关键符号）。监管者交叉验证不通过 = 该格禁止填写。**

派发与验收流程见第 5 节 MMO 任务板调度。

#### 第四步：全部完成后核对矩阵

全部文件审计完成后，检查矩阵中无空列、无空行。输出最终确认：

```
总计: N 个文件 × M 视角 = N×M 次审查，全部完成，0 遗漏，0 空列
```

#### 铁律

- "核心文件"、"重要文件"、"入口文件" 这些概念**不存在**。每一个文件都是核心文件。
- "看起来差不多"、"功能类似"、"命名相近" → 不可以跳过，每个文件名不同就必须单独读。
- 用户指定的目录范围是硬边界——说 `app/` 就是 `app/` 下全部文件，一个不漏。
- 监管者不代替 Agent 读文件——Agent 必须自己 Read，监管者只交叉验证 Agent 返回的证据（内容见证/行数/关键符号）是否与源文件一致。
- **禁止参考任何文档、README——只认代码。代码就是唯一的真相。**

### 2.5 七层审查框架（每个 Agent 强制执行）

审查一个文件时，必须按以下顺序自底向上逐层检查（下面序号小的是“㡳”）。不可跳跃、不可颠倒、不可反向：

```
1. 代码行
2. 函数
3. 文件
4. 功能
5. 模块
6. 架构
7. 整体
```

框架只规定层次顺序，各视角在自己的检查项中定义每层审什么。

**"代码行""函数""文件"三层为强制层**——每个 Agent 审查任何一个文件都必须走完这三层。"功能"及以上的层可能跨单文件边界，此时审查本文件所涉及的部分并注明范围。

**每层输出必须注明审查范围（行号）**，格式：`L{层名}: [发现或"审查完成，未发现异常"] — :行号范围`

### 2.6 ATAM 风险分类体系

| 分类 | 定义 | 标记 |
|------|------|:---:|
| **风险 (Risk)** | 可能导致质量属性不达标的架构决策 | ⚠️ |
| **非风险 (Non-Risk)** | 有助于质量属性达标的正确决策 | ✅ |
| **敏感性点 (Sensitivity Point)** | 微调即显著影响某一质量属性的决策参数 | 🔸 |
| **权衡点 (Tradeoff Point)** | 同时影响多个质量属性、需要取舍的决策 | ⚖️ |

Agent 发现任何问题必须标注以上 ATAM 分类。

**ATAM 与问题等级的关系**：两套体系分层使用，不互替代。

| 体系 | 阶段 | 回答的问题 | 位置 |
|------|------|-----------|------|
| ATAM | 审计中 | 这是什么性质？ | §2.6 |
| P0-P3 | 审计后汇总 | 先修哪个？ | §8.1 |

同一个发现同时有两层标签：例如"用户数据纯内存 = ATAM:风险 + P0:致命"。Agent 审计中只标 ATAM，汇总阶段再定 P0-P3 和 safe-to-auto-fix/needs-human。

### 2.7 架构评估方法

**ATAM（Architecture Tradeoff Analysis Method, SEI/CMU）** — 多角色评估 + 场景化质量属性评估，用于对系统架构进行结构化权衡分析。

### 2.8 审查记录规范

**`[✓]` 标记必须附代码级证据才能算完成审查。**

#### 行数必须精确

读到文件末尾，记录文件的实际总行数。行数来自 Read 工具返回最后一行的行号，禁止估算。

#### 每个文件必须提取关键符号

读完一个文件后，输出该文件中定义的至少 3 个关键符号（函数名/类名/变量名）。如果文件少于 3 个符号（如 `__init__.py`、纯配置文件），列出所有符号。符号名必须精确匹配代码中的实际命名。

#### 每个视角必须输出审查记录

对每个文件应用每个视角后，必须输出审查记录。分两种情况：

**发现了问题**：
```
视角X（视角名）: ⚠ [问题描述] — :行号
```

**未发现问题**——必须写清审了什么、审了多少行：
```
视角X（视角名）: 逐行审查完成，[检查了哪些维度]，未发现异常 — :1-193
```

不满足要求的写法：
- `视角X: 无问题` — 不知道审了什么、审了多少
- `视角X: 通过` 或 `视角X: OK` — 空无一物
- 空着不填

#### 交叉验证

任意两个文件的审查记录不能完全一样。如果两个文件内容确实相同（如均为空 `__init__.py`），必须在记录中明确说明。

#### 要求

- 一个文件缺少任一视角的记录 = 审计未完成
- 一条记录缺少行号 = 该记录不完整
- N 个文件 × M 视角 = N×M 条记录，少一条都不行

## 3. 视角系统

### 3.1 核心视角（8个，默认全部启用）

适用于所有软件项目，无需任何配置：

| # | 视角 |
|:---:|------|
| V1 | 工程架构 |
| V2 | 产品市场 |
| V3 | 质量测试 |
| V4 | AI内容审计 |
| V5 | 项目自我审视 |
| V6 | 跨角色整合 |
| V7 | 安全合规 |
| V8 | 性能工程 |

各视角的具体检查项以 `references/perspectives/v{n}-*.md` 为准，Agent 必须逐项检查，不可遗漏。

**V4与V5区分**：V4 是代码级——逐行检测注释/docstring中的AI生成痕迹，对每条原子主张做代码验证；V5 是项目级——从整体判断项目宣称优势的真实性。V5 的三类优势分类引用 V4 的代码级发现，自身不做AI文本特征扫描。AI语言审计统一归 V4，竞品对比验证统一归 V2（产品市场）。

### 3.2 领域视角（自动检测启用）

**不依赖用户手动配置。** 在文件扫描阶段，每读一个文件，自动检测代码内容中是否包含领域关键词。命中任意一个关键词，该领域视角即自动启用。

| # | 领域视角 | 检测关键词 |
|:---:|------|---------|
| D1 | 数据工程 | ETL, pipeline, data-warehouse, spark, kafka, airflow, 数据管道, 数据仓库, 数据湖, 流处理, 批处理, data-lake, lakehouse, delta-lake, iceberg, parquet, ORC, flink, beam, dagster, dbt, data-quality, schema, partition, backfill, lineage, ingestion, streaming, batch, OLAP, OLTP, CDC, data-mesh, medallion, exactly-once, idempotent, watermark, 数据质量, 数据血缘, 数据治理, 回填, 幂等, 水位线 |

**检测实现**：

```
监管者在 Step 2 扫描文件清单后:
  1. 抽样 Read 若干文件内容（至少覆盖不同目录，确保检测覆盖面）
  2. 对每个未启用的领域视角，搜索其关键词列表
  3. 命中 -> 该领域视角启用，输出提示: "[领域视角] D1 数据工程已自动启用（在 {file_path} 中检测到关键词 {keyword}）"
  4. 全部抽样文件扫描完仍未命中 → 该领域视角不启用，对应 Agent 不派发
```

## 5. MMO 任务板调度机制

### 5.1 概念映射

```
公会会长 (Guild Master)  =  Supervisor（监管）
冒险者 (Adventurer)       =  视角 Agent（每个 Agent 专精一个视角，审全部文件）
任务板 (Quest Board)      =  文件 x 视角矩阵 + .audit-quest-board.json
任务 (Quest)              =  一个视角 x 全量文件（矩阵一整列）
任务书 (Quest Scroll)     =  Agent 收到的派发指令包（含全量文件列表）
领取任务                  =  会长一次性并行派发所有任务给 N 个 Agent
做任务                    =  Agent 逐文件 Read + 七层审查
交任务                    =  Agent 返回全部文件的结构化证据记录
会长验收                  =  Supervisor 逐文件逐项核对
任务驳回 / 重接            =  整列退回重审，换同视角新 Agent，最多3次
任务失败                  =  3次驳回，该列所有格标记"需人工审查"
```

### 5.2 任务板 = 文件 x 视角矩阵

这是整个调度系统的唯一数据源。**每一列（一个视角）= 一个 Agent 负责**：

```
                    视角 ->
文件      V1      V2      V3      V4      V5      V6      D1
f_001     [ ]     [ ]     [ ]     [ ]     [ ]     [ ]     [ ]
f_002     [ ]     [ ]     [ ]     [ ]     [ ]     [ ]     [ ]
f_003     [ ]     [ ]     [ ]     [ ]     [ ]     [ ]     [ ]
...
         ^^^^^^^ Agent-V1 负责此列全部文件 ^^^^^^^
```

每个格是一个审查单元。一个 Agent 负责一整列（一个视角 × 全量文件），审完后填满该列所有格。

### 5.3 任务状态机（5态，以列为单位）

一个任务 = 一个视角列（一个 Agent 负责全量文件）。状态机追踪整列进度：

```
  [可领取]  ----  初始 / 驳回后重置（attempts < 3）
      |
      | 会长派发给 Agent（含全量文件列表）
      v
  [进行中]  ----  Agent 逐文件审查中
      |
      | Agent 返回全部文件审查结果
      v
  [已提交]
      |
      | 会长逐文件逐项验证
      |
  ----+----
  |         |
  v         v
[已完成]  [可领取]  ----  attempts < 3，换 Agent 重派整列
              |
              | attempts = 3
              v
          [已失败]  ---->  报告标记"需人工审查"，该列所有格标记为需人工审查
```

### 5.4 Agent 派发（会长实际操作）

**Step A — 构造 Agent prompt**。会长将以下三部分拼接为一个完整 prompt：
1. **通用纪律** — 本文 §2.5（七层框架）+ §2.6（ATAM）+ §2.8（审查规范）+ §5.7（交叉验证）
2. **视角指令** — `references/perspectives/v{n}-*.md` 完整内容（如启用领域视角则追加 `references/domains/d{n}-*.md`）
3. **当前任务** — 全量文件列表 + 视角名

**Step B — 调用 Agent 工具派发**（所有视角 Agent 一次性并行派发）：
```
Agent({
  subagent_type: "general-purpose",
  description: "审查全部文件 {视角名}",
  prompt: "{Step A 拼接好的完整prompt}"
})
```

会长不传代码内容——只传文件路径列表。每个 Agent 必须自己逐文件 Read，返回的行数/内容见证/关键符号与源文件交叉比对一致才算验收通过（§5.6）。

**Agent 收到的完整 prompt 模板**（会长据此拼接）：

```
你是{视角名}审查专家，专精此一个视角。

## 审查纪律（通用）
{§2.5 七层框架内容}
{§2.6 ATAM分类体系}
{§2.8 审查记录规范}
{§5.7 交叉验证机制}

## 视角指令
{references/perspectives/v{n}-*.md 完整内容}

## 当前任务
视角: {视角名}
文件列表: 共 F 个文件
```
{file_1}
{file_2}
...
{file_F}
```

## 审查完整性要求
1. 按文件列表顺序，逐文件调用 Read 读取
2. 每读完一个文件立刻 date 获取时间戳
3. 每文件按七层审查框架自底向上审查，不可跳跃
4. 每文件返回时必须包含:
   - 行数 N（Read 返回最后一行行号）
   - 时间戳（格式 YYYY-MM-DD HH:MM:SS）
   - 内容见证（第 floor(N/2) 行原文逐字抄录）
   - 至少 3 个实际存在的关键符号
   - 每层审查结果 + 行号范围
   - 发现的问题标注 ATAM 分类 + 违反的代码规范编号
5. "代码行""函数""文件"三层不可跳过
6. 对代码中违反行业规范之处，标注具体规则编号

全部文件审查完成后，汇总返回所有文件的审查结果，格式见下方交任务格式。
```

### 5.5 Agent 交任务格式

一个 Agent 返回该视角下**全部文件**的审查结果：

```
Agent-V1 交任务 #001（视角: 工程架构，共 F 个文件）:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
文件 #1: app/api/server.py (67行)
读取时间: 2026-06-09 22:31:05 | 见证: L33: app = FastAPI(title="MyApp")
关键符号: create_app, health_check, api_router

### 5.6 会长验收清单

验收范围为一个 Agent 返回的全部文件。逐文件逐项核对：

```
验收 Agent-V1（视角: 工程架构，共 F 个文件）:
------------------------------------------------
逐文件检查:
[1] 每文件覆盖七层（至少"代码行""函数""文件"三层）？   [通过/驳回]
[2] 每文件每层有行号/审查范围？                        [通过/驳回]
[3] 每文件关键符号在源文件中实际存在？                  [通过/驳回]
[4] 每文件内容见证与源文件第 N/2 行原文一致？           [通过/驳回]
[5] 每文件审查结果非空（每层至少一句描述）？             [通过/驳回]
[6] 时间戳与 Read 调用在秒级间隔内？                    [通过/驳回]
------------------------------------------------
验收结论: [通过/驳回]
如有驳回，标注具体文件+具体条目+原因
```

驳回时整列退回重审（换同视角新 Agent），attempts++。

### 5.7 交叉验证机制

Agent 不读文件就无法通过验收：

| Agent 声称 | 会长验证方式 |
|-----------|------------|
| 文件有 N 行 | Read 返回的最末行号 = N？ |
| L_{N/2} 行内容是 XXX | 查看源文件该行原文 |
| 关键符号 A、B、C 存在 | 搜索源文件 |
| L函数审查了 :49-50 | 行号在 1~N 范围内？描述的函数确实在49-50行？ |
| 时间戳是 T | 对话日志中有对应的 Read 调用时刻 |

核心逻辑：没读文件 -> 不知道 N/2 行写什么 -> content_witness 编不出来 -> 不匹配 -> 驳回。不依赖信任，只依赖交叉比对。

### 5.8 调度节奏

以视角为波次（N 个视角 Agent 一次性并行派发，各审全部文件）：

```
会长一次性并行派发 N 个 Agent（N = 核心视角数 + 已启用领域视角数），每个 Agent 收到全量文件列表。
在同一轮回复中并行调用 N 次 Agent 工具（每条 subagent_type: "general-purpose"，prompt 含完整文件列表）。

各 Agent 内部按文件列表顺序逐文件审查，审完一个接下一个，无需等待其他 Agent。
全部 Agent 返回后 -> 会长逐文件逐视角验收 -> 填满整个矩阵

...

全部波次完成后:

回收站清理:
  收集所有被驳回的视角列 -> 换同视角新 Agent 重派整列
  通过 -> [已完成]
  仍被驳回且 attempts = 3 -> [已失败]

最终:
  [已完成]: 6/7 视角列
  [已失败]: 1/7 视角列 -> 报告标记"需人工审查"
```

### 6.2 断点续做

审计中断后（会话关闭 / 进程崩溃 / 用户手动停止），重新启动审计时：

```

## 7. 跨角色整合（5 + N 角色体系）

### 7.1 通用核心角色（5个，V6 始终包含）

| # | 角色 | 关注点 | 对应视角 |
|:---:|------|------|---------|
| R1 | 终端用户 | 界面响应、错误提示友好度、操作连贯性、崩溃恢复 | V1 + V2 |
| R2 | 开发者/维护者 | 代码可维护性、架构耦合、注释一致性、测试覆盖 | V1 + V3 |
| R3 | 产品/业务方 | 功能完整度、与产品定位一致性、竞品差异化的代码支撑 | V2 + V5 |
| R4 | 运营/SRE | 部署可靠性、监控完备度、日志可追溯、灾备与熔断 | V1 + V3 |
| R5 | 安全/合规方 | 权限模型、数据加密、审计追溯、OWASP 覆盖 | V1 + V3 + V7 |

### 7.2 领域角色（由领域视角追加）

以数据工程领域视角（D1）为例，启用后追加：

| # | 角色 | 关注点 |
|:---:|------|------|
| R6 | 数据工程师 | 数据管道可靠性、数据质量监控、Schema 演进兼容性 |
| R7 | 数据架构师 | 存储选型合理性、数仓分层规范、数据血缘完整性 |

### 7.3 V6 工作方式

V6（跨角色整合）Agent 不直接审查代码，而是汇总同文件其他视角 Agent 的发现，从各角色视角重新组织问题。每个问题关联到具体代码证据（文件:行号）。对于角色关注点在本文件中无对应发现的情况，注明"本文件未涉及该角色的相关代码"。

**V6 依赖**：V6 Agent 必须在所有其他视角 Agent 返回并通过验收后派发，因为它汇总其他视角的审查结果。

## 8. 自动修复 + 回归检测 + 交互式 Walkthrough

### 8.1 问题分类

审查完成后，所有发现的问题按两个维度标注（ATAM 定性见 §2.6，此处定级定分流）：

**风险等级**：

| 等级 | 定义 | 示例 |
|------|------|------|
| P0 | 致命：数据丢失/安全风险 | 用户数据纯内存、CORS全开、密钥硬编码 |
| P1 | 影响正确性/可信度 | 数据源无校验、关键操作无审计日志 |
| P2 | 架构债务/功能缺失 | 缺少统一错误处理、测试覆盖不足 |
| P3 | 长期优化 | float 改 Decimal、性能优化 |

**修复安全性**：

| 标记 | 定义 | 示例 |
|------|------|------|
| safe-to-auto-fix | 纯配置/增量写入/新组件，不改现有逻辑 | CORS限定、新增组件、格式化 |
| needs-human | 涉及算法逻辑/状态机/核心流程 | 权限规则修改、引擎替换 |

### 8.2 自动修复（safe-to-auto-fix 项）

```
Step 1: 自动修复
  1. 筛选 safe-to-auto-fix 项，按模块分组
  2. 修复顺序: 配置文件 -> 基础设施（新增组件/表） -> 现有逻辑
  3. 每条修复单独 commit
  4. 修复后重跑受影响文件的对应视角 Agent 审查
  5. 确认问题已消除（对应视角列从"发现问题"变为"审查完成，未发现异常"）
```

### 8.3 回归检测

```
Step 2: 回归检测
  1. 对比修复前后的审计矩阵
  2. 确认:
     (a) 修复项对应的视角列从"发现问题"变为"审查完成，未发现异常"
     (b) 其他视角列未引入新的问题发现
  3. 如有新问题 -> 标记来源 commit -> 回滚或修复
```

### 8.4 交互式 Walkthrough（needs-human 项）

```
Step 3: 交互式 Walkthrough
  对每条 needs-human 问题，逐条展示:
    - 发现描述 + 代码证据（文件:行号）
    - 修复建议 + 预估影响范围（受影响文件数、模块数）
    - 用户决策: [修] / [跳过] / [标记为已知风险]
```

## 12. 完整执行流程（7步）

```
Step 1 -- 加载配置，确定初始视角
  读 audit-config.yaml（如有） -> 确定核心视角列表 -> 加载对应指令文件
  领域视角初始为空，等待 Step 3 自动检测

Step 2 -- 扫描代码库，生成任务板
  Glob/codegraph_files 扫描全部文件 -> 排除配置目录 ->
  生成 N文件 x M视角 任务矩阵 -> 全部 [可领取]
  创建 .audit-quest-board.json

Step 3 -- 创建报告，写入空矩阵
  列: # | 文件 | 读取 | 时间戳 | 内容见证 | V1 | V2 | ... | 关键符号
  所有单元格初始为空
  列数 = 1(序号) + 1(文件) + 1(读取) + 1(时间戳) + 1(内容见证) + M(视角数) + 1(关键符号)

Step 4 -- 一次性派发全部 Agent（N 视角并行，各审全量文件）
  ① 监管扫描全部文件 -> 生成完整文件清单
     **同时扫描领域关键词** -> 命中则启用对应领域视角 -> 追加到视角列表
  ② 会长发布 N 个任务到任务板（N = 当前已启用视角数）
  ③ 一次性并行派发 N 个 Agent：在同一轮回复中并行调用 N 次 Agent 工具（subagent_type: "general-purpose"），每人一份拼接好的完整 prompt（§5.4），含全量文件列表 + 通用纪律 + 视角指令
     [可领取] -> [进行中]
  ④ 每个 Agent 按文件列表顺序逐文件 Read -> 七层审查 -> 全部审完后汇总返回
     [进行中] -> [已提交]
  ⑤ 会长逐文件逐视角验收（验证清单见 5.6）
     通过 -> [已完成]，填矩阵对应行对应列
     驳回 -> [可领取]，attempts++，记录驳回原因
  ⑥ 全部验收完成后更新 .audit-quest-board.json
  ⑦ 输出任务板最终进度

Step 5 -- 重派回收站
  收集被驳回任务 -> 换 Agent 重派（同一视角不同实例）->
  attempts < 3 的继续尝试 -> attempts = 3 -> [已失败]，标记"需人工审查"

Step 6 -- 生成报告
  核对矩阵无空列 -> 交叉比对 -> 按模板生成六章报告 ->
  声明: N文件 x M视角 = NxM 次审查，0遗漏

Step 7 -- 自动修复 & Walkthrough
  safe-to-auto-fix -> 自动修 + commit + 回归检测
  needs-human -> 交互式逐条决策
```
