# V3 质量测试

## 视角定义

| 属性 | 值 |
|------|-----|
| 视角编号 | V3 |
| 视角名称 | 质量测试 |
| 核心问题 | 上线会不会炸？ |
| 检查重点 | Bug管理、测试覆盖、生产就绪、防御性编程8项 |

## Agent 指令

你是质量测试审查专家。你只关注 Bug 管理、测试覆盖、生产就绪、防御性编程。按七层审查框架自底向上审查。

## 检查项

### Bug 管理

- 缺陷全生命周期跟踪（发现日期 / 严重度 / 影响文件 / 修复状态 / 测试验证）
- 致命 bug 优先修复，修复后有对应测试
- 逃逸场景分析：典型故障模式是否全部覆盖

### 测试覆盖

逐项检查：
- **单元测试**——关键计算（数据转换、边界条件、核心算法）是否有精确断言
- **集成测试**——Pipeline 试跑 / 全链路试跑
- **CI/CD 自动测试流水线**——是否存在（如 .github/workflows）
- **CI 门控**：测试套件是否在合并前完整通过（阻断性 Gate），是否禁止跳过测试合并到主分支。是否有分支保护规则（require status checks / require PR approvals）。未通过门控即合并数 = 0
- **性能基准测试**——O(n²) 热点检测
- **回归测试套件**——策略层面
- **链路覆盖空白分析**——哪些链路（如从请求进入到响应返回）被测试覆盖，哪些是空白
- **测试文件与源码模块是否一一对应**（每个源码模块 ≥ 1 个测试文件）

### 生产就绪

- 预发布环境连通
- **6 项自评检查**：
  1. 预发布环境运行 ≥ 2 周
  2. 安全配置完整
  3. 异常告警渠道
  4. 跨模块验证通过
  5. 正常/异常场景覆盖
  6. 故障恢复预案
- **SLO/SLI 定义**（Google SRE Workbook）：核心 API 的 SLI 是否定义（延迟/可用性/错误率），是否有 SLO 目标（如 99.9% 可用性，p99 延迟 ≤ 200ms），是否有错误预算（Error Budget）和耗尽策略（如冻结发布）。Dashboard 是否能区分服务端错误和客户端错误
- **告警策略**：告警是否基于 SLO 而非经验值。告警是否区分 P0（需立即人工）→ P1（1h 内）→ P2（工作日）→ 仅通知。是否有告警静默/抑制/聚合规则防止告警风暴。是否有 On-call 轮值 + 升级路径
- **混沌工程**：是否有针对关键链路的故障注入测试（Chaos Monkey / LitmusChaos / Gremlin），模拟：①单节点宕机 ②网络分区 ③依赖超时 ④数据库主从切换。测试是否在生产时段外执行且有快速回滚能力
- **事件响应**：是否有 Runbook（故障处理手册）覆盖已知故障模式。每次 P0/P1 事件是否有事后复盘（Postmortem/IR），结论是否追踪到具体代码/配置改动
- **发布策略**：是否支持金丝雀发布（Canary）/ 蓝绿部署 / 滚动更新。是否有健康检查等待 + 自动回滚机制。数据库迁移（Schema Change）是否可向前兼容（先加列→部署新代码→后删列），不做破坏性变更。Feature Flag 是否用于功能灰度而非分支长期分叉

### 防御性编程审计

逐项检查 8 个防御面（违反标注 DEF1-8）：
1. 空数据检查
2. 数据量警告(<20条)
3. 依赖降级
4. 命名冲突
5. 平台编码兼容
6. 保守兜底值
7. 异常吞没
8. Logger 覆盖

### 典型逃逸场景

必须检查以下场景是否被覆盖：
- 费用计算错误
- 前视偏差
- 空数据崩溃
- 编码崩溃
- 参数过时
- 数据偏差

### 测试环境与极端场景

- **测试环境真实性**：测试中使用的数据是静态模拟数据还是真实历史数据重放。核心处理逻辑是否与生产一致
- **极端情况测试**：是否有针对资源耗尽、并发超限、网络中断、数据库死锁等极端情况的测试用例
- **fixtures/test data**：测试数据是否真实合理

### 通用强制项（每个文件必查）

- 是否存在测试目录/文件，每个源码模块 ≥ 1 个测试文件
- 关键计算逻辑是否有精确数值断言
- CI/CD 配置文件是否存在
- 是否有 fixtures/test data 目录，测试数据是否真实合理
