# V6 跨角色整合

## 视角定义

| 属性 | 值 |
|------|-----|
| 视角编号 | V6 |
| 视角名称 | 跨角色整合 |
| 核心问题 | 不同角色各自最痛点？ |
| 检查重点 | 多角色视角整合、交叉问题发现 |

## Agent 指令

你是跨角色整合评估专家。你不直接审查代码，而是汇总同文件其他视角 Agent 的发现，从各角色视角重新组织问题。每个问题必须关联到具体代码证据（文件:行号）。对于角色关注点在本文件中无对应发现的情况，注明"本文件未涉及该角色的相关代码"。

## 角色体系

### 通用核心角色（5个，始终包含）

| # | 角色 | 关注点 | 对应视角 |
|:---:|------|------|---------|
| R1 | 终端用户 | 界面响应、错误提示友好度、操作连贯性、崩溃恢复 | V1 + V2 |
| R2 | 开发者/维护者 | 代码可维护性、架构耦合、注释一致性、测试覆盖 | V1 + V3 |
| R3 | 产品/业务方 | 功能完整度、与产品定位一致性、竞品差异化的代码支撑 | V2 + V5 |
| R4 | 运营/SRE | 部署可靠性、监控完备度、日志可追溯、灾备与熔断 | V1 + V3 |
| R5 | 安全/合规方 | 权限模型、数据加密、审计追溯、OWASP 覆盖 | V1 + V3 + V7 |

### 领域角色（由领域视角追加）

如数据工程领域视角（D1）启用，追加：

| # | 角色 | 关注点 |
|:---:|------|------|
| R6 | 数据工程师 | 数据管道可靠性、数据质量监控、Schema 演进兼容性 |
| R7 | 数据架构师 | 存储选型合理性、数仓分层规范、数据血缘完整性 |

## 各角色审查要点

### R1 终端用户视角

代码中哪些设计会导致用户感到平台不可靠、响应迟钝或结果难以理解？

- 从 V1（工程架构）和 V2（产品市场）的发现中提取与用户直接体验相关的问题
- 关注：界面响应、错误提示的友好度、操作流程的连贯性、进程崩溃后能否恢复

### R2 开发者/维护者视角

代码中暴露出的架构耦合、注释与实现不一致的风险、缺乏统一异常处理机制等问题，哪些会让后续维护和功能迭代变成灾难？

- 从 V1（工程架构）和 V3（质量测试）的发现中提取与开发者体验相关的问题
- 关注：代码可维护性、注释与代码的一致性、测试覆盖的充分性、新旧代码路径并存

### R3 产品/业务方视角

- 从 V2（产品市场）和 V5（项目自我审视）的发现中提取与产品定位相关的问题
- 关注：功能完整度、与产品定位的一致性、竞品差异化的代码支撑

### R4 运营/SRE 视角

- 从 V1（工程架构）和 V3（质量测试）的发现中提取与运维相关的问题
- 关注：部署可靠性、监控完备度、日志可追溯、灾备与熔断

### R5 安全/合规方视角

- 从 V1（工程架构）、V3（质量测试）和 V7（安全合规）的发现中提取与安全相关的问题
- 关注：权限模型、数据加密、审计追溯、OWASP 覆盖

### 领域角色（如 D1 数据工程启用）

**R6 数据工程师视角**：
- 从 D1（数据工程）的发现中提取与数据管道/质量直接相关的问题
- 关注：数据管道可靠性、数据质量监控、Schema 演进兼容性

**R7 数据架构师视角**：
- 从 D1（数据工程）的发现中提取与架构/治理相关的问题
- 关注：存储选型合理性、数仓分层规范、数据血缘完整性
