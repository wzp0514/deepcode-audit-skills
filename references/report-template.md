# 审计报告模板

---

## 报告输出位置

**审计报告必须生成在用户当前项目的 `docs/` 目录下**（即 `{项目根目录}/docs/系统审计报告-YYYY-MM-DD.md`）。

## 报告命名规范

每次审计生成的报告文件，使用中文名 + 日期（日级精度）：

```
系统审计报告-YYYY-MM-DD.md
```

## 报告头（每次审计必填）

```markdown
# 系统深度审计报告

**审计时间**：YYYY-MM-DD HH:MM:SS
```

审计时间取当前系统真实时间，通过 `date` 命令获取，禁止编造。除此之外报告头不包含任何其他信息。

## 第一章：审计范围与文件清单确认

- 列出代码库根路径、项目技术栈（语言/框架/数据库等，从代码文件中实际读取，不从文档推断）。
- **输出完整文件清单表格**（审计开始前生成空矩阵，逐文件闭环填入）：

```
| # | 文件 | 读取 | 时间戳 | 内容见证 | V1工程 | V2产品 | V3质量 | V4 AI | V5审视 | V6跨角色 | (领域视角如有) | 关键符号 |
|---|------|------|--------|---------|--------|--------|--------|-------|--------|---------|---------------|---------|
| 1 | app/api/server.py | 待读 | | | | | | | | | | |
| 2 | app/service/user_mgr.py | 待读 | | | | | | | | | | |
```

**每列的含义和填写规则：**

**"读取"列** — 文件是否已被 Read 工具实际读取。合法值：
- `待读` — 尚未读取
- `已读(N行)` — 已逐行读取完毕，N 为 Read 工具返回最后一行的行号

**"时间戳"列** — Read 返回后立即调用 `date` 获取的时间（`YYYY-MM-DD HH:MM:SS`）。该时间戳必须能对应到本轮对话中的一次 Read 调用，且与 Read 调用间隔在秒级。读取列=待读时为空。

**"内容见证"列** — 从 Read 返回内容中截取的第 `floor(N/2)` 行原文，逐字抄录（含缩进）。格式：`L96: self.users: list[dict] = []`。这一列的内容无法凭记忆或推断得出——没读文件就不可能知道中间某行写了什么。读取列=待读时为空。

**视角列** — 该视角是否已对该文件完成审查。合法值：
- 空 — 尚未审查
- `已审: [具体发现] — :行号` — 已完成审查，附发现描述和代码行号引用

核心视角列始终包含 V1~V6。领域视角列（如 D1 数据工程）在自动检测启用后动态追加。

**"关键符号"列** — 从 Read 返回内容中提取。读取前为空：
- 空 — 文件尚未读取
- `函数A, 类B, 变量C` — 该文件中实际定义的至少 3 个符号名

**阶段门控规则（逐行强制）：**

```
读取列 = "待读"  →  时间戳/内容见证/关键符号/视角列 必须全部为空
读取列 = "已读(N行)"  →  时间戳/内容见证/关键符号 必须已填写，视角列可为空或已审
任一视角列 = "已审: ..."  →  读取列必须已是"已读(N行)"，时间戳+内容见证必须已有
全部视角列 = "已审: ..."  →  该文件审计完成
```

**可验证性检查（事后审计对话日志时可逐条核对）：**
- 时间戳与对话日志中 Read 调用时刻不一致 → 编的
- 内容见证的第 N/2 行与实际文件不符 → 没读就填的
- 关键符号在当前代码中不存在 → 凭记忆编的
- 视角记录行号引用的内容与描述矛盾 → 推断的，不是逐行读的

- 如某领域无对应代码文件（如无测试文件、无 CI 配置），明确注明：「[领域] 无对应代码文件，该领域无法评估」。

## 第二章：逐文件 Agent 审计记录

> 对每个文件，全部已启用视角的 Agent 并行审查，每个 Agent 按七层自底向上输出结果。监管验证通过后填入矩阵。

### 单文件 Agent 输出格式

对每个文件，各 Agent 各输出一份按层组织的审查记录：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
文件 #N: app/service/user_mgr.py (193行)
读取时间: 2026-06-09 21:30:06 | 见证: L96: self.users: list[dict] = []
符号: UserManager, create_user, update, delete, _find
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Agent-V1 [工程架构]:
  L代码行: 逐行审查完成，无硬编码、无不安全写法 — :1-193
  L函数: UserManager.__init__ 直接初始化list无持久化 — :49-50
    ⚠ 风险: 用户数据纯内存，进程崩溃丢失全部数据 — :50
  L文件: 单一职责(用户生命周期)，依赖方向正确 — :37-49
  L功能: 用户创建→更新→删除流程完整 — :53-166
  L模块: 位于app/service/，模块边界清晰
  L架构: 符合分层架构，位于服务层
  L整体: 无跨模块循环依赖，但缺持久化影响可靠性

Agent-V2 [产品市场]:
  L代码行: 逐行审查完成，命名清晰 — :1-193
  L函数: create_user参数文档完善 — :53-101
  L文件: API设计直观(create/update/delete/get)，用户友好
  L功能: 用户状态机对调用方透明
  L模块~L整体: 本文件内审查完成，未发现异常

... (remaining agents)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

监管验证: ✅ 全部 M 份通过
```

**每层输出格式**：
- `L{层名}: [审查结果] — :行号范围`
- 发现问题标注 `⚠ 风险:` / `⚠ 敏感性点:` / `⚠ 权衡点:`
- "代码行""函数""文件"三层为强制层，不可跳过
- "功能""模块""架构""整体"跨文件层如超出本文件范围，注明审查了本文件涉及的哪些部分

**以下 Agent 输出不满足要求，监管应驳回**：
- 缺"代码行""函数""文件"任一层 → 驳回
- 任一层无行号范围 → 驳回
- 所有层全是"未发现异常"且审查范围为空 → 驳回
- 两个 Agent 返回逐字相同 → 驳回

## 第三章：跨角色视角整合评估

基于第二章的发现，分别从各角色的视角重新组织问题。每个问题必须关联到具体代码证据（文件:行号）。

### 通用核心角色

#### 终端用户视角（R1）

代码中哪些设计会导致用户感到平台不可靠、响应迟钝或结果难以理解？

#### 开发者/维护者视角（R2）

代码中暴露出的架构耦合、缺乏统一异常处理机制等问题，哪些会让后续维护和功能迭代变成灾难？

#### 产品/业务方视角（R3）

- 关注：功能完整度、与产品定位的一致性、竞品差异化的代码支撑

#### 运营/SRE 视角（R4）

- 关注：部署可靠性、监控完备度、日志可追溯、灾备与熔断

#### 安全/合规方视角（R5）

- 关注：权限模型、数据加密、审计追溯、OWASP 覆盖

### 领域角色（如 D1 数据工程启用）

#### 数据工程师视角（R6）

数据管道是否可靠且可观测？数据质量监控是否完善？Schema 变更是否向前兼容？

#### 数据架构师视角（R7）

存储选型是否合理？数仓分层是否规范？数据血缘是否完整可追溯？

## 第四章：修正后的架构优化路线图

结合审计发现（第二章+第三章），给出分阶段优化路线图。

### 架构层（宏观）

**P0 致命阻碍（必须立即解决才能进入下一阶段）**：
- 列出会导致资金风险、数据永久损坏、不可逆的架构锁死的具体问题（引用出处），并给出解决思路。

**P1 核心功能正确性与可信度修复（1-3个月）**：
- 针对业务逻辑的修补与重构计划。

**P2 可维护性与体验升级（3-6个月）**：
- 解耦、测试体系建设、完整审计日志、可视化及交互优化等。

每条路线图任务都应能从审计结果中找到直接依据。

### 代码层（微观）

> 仅在架构层面无 P0 致命问题后展开。若存在 P0 问题，注明「架构存在致命阻碍需优先解决，代码层修改暂不细化。」

按模块组织，逐级细化：

```
模块名：
  文件路径:行号 → 问题描述 → 修改建议 → 预期效果
  文件路径:行号 → 问题描述 → 修改建议 → 预期效果
  ...
```

从架构宏观（P0/P1/P2 方向）→ 模块中观（模块级重构）→ 代码行微观（具体修改），逐级细化，不可跳跃。

## 第五章：问题汇总与优先级排序

> 汇总全量审计中所有发现的问题，按风险等级排序，标注修复安全性和预估影响范围。

### 5.1 总览

```
审计范围: N 个文件
发现问题: M 条
  P0（致命）:    X 条
  P1（影响正确性）: Y 条
  P2（架构债务）:   Z 条
  P3（长期优化）:   W 条

可自动修复（safe-to-auto-fix）: A 条
需人工决策（needs-human）:     B 条
```

### 5.2 按优先级排序的问题清单

```
| # | 等级 | 文件:行号 | 问题描述 | 视角 | 修复安全性 | 修复建议 |
|---|------|----------|---------|------|-----------|---------|
| 1 | P0 | user_mgr.py:50 | 用户数据纯内存，进程崩溃丢失 | V1工程 | safe-to-auto-fix | SQLite持久化 |
| 2 | P0 | server.py:67 | CORS全开+无认证 | V1工程 | safe-to-auto-fix | 限定origin+API Key |
| 3 | P1 | data_fetch.py:112 | 数据源无校验 | V3质量 | safe-to-auto-fix | 添加校验 |
| 4 | P1 | auth/engine.py:358 | 权限规则硬编码 | V3质量 | needs-human | 配置化需重新设计 |
| ... | | | | | | |
```

**表格填写规则**：
- 按等级排序（P0→P1→P2→P3），同等级内按影响范围排序（影响文件数多的排前面）
- 每条问题必须附文件:行号 + 视角归属
- 修复安全性只能是 `safe-to-auto-fix` 或 `needs-human`，不可留空
- P0 且 `safe-to-auto-fix` 的问题标记为首批自动修复

### 5.3 自动修复执行计划（safe-to-auto-fix 项）

列出所有 `safe-to-auto-fix` 项，按模块分组，标注每条修复对应的文件、行号、修改内容。修复执行顺序：先改配置文件 → 再改基础设施（新增组件/表） → 最后改现有逻辑。

### 5.4 待人工决策清单（needs-human 项）

逐条列出，格式：
```
#N [P1] auth/engine.py:358 — 权限规则硬编码if-else链
  当前: 9个检查顺序硬编码
  建议: 改为配置驱动(YAML定义规则链)，支持动态增减
  影响: app/service/auth/engine.py + settings.yaml + 所有调用模块(约5个)
  决策: [ ] 修  [ ] 跳过  [ ] 标记为已知风险
```

## 第六章：最终审计声明

> 本人工智能模型，以系统架构审计专家角色确认：本次审计已无遗漏地覆盖了代码库全部 [N] 个文件，逐文件逐行审查，应用全部 [M] 个视角（含 [X] 个核心视角 + [Y] 个领域视角）对每个文件进行了 M×N 次审查，0 文件遗漏、0 视角跳过。审计过程中未引入任何未在代码中实际存在的主观臆断或外部知识补充。发现 [Z] 条问题（P0:[p0] / P1:[p1] / P2:[p2] / P3:[p3]），其中 [A] 条可自动修复、[B] 条需人工决策。所有审计意见均严格锚定于代码实现（文件:行号）。
